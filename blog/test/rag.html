What is RAG?

A standard LLM can only use what it memorised during training. If you ask it about a document it never saw, it either hallucinates or says "I don't know."

Retrieval-Augmented Generation (RAG) gives the LLM an external, searchable memory. Before the model generates an answer, a retriever finds the most relevant passages from a document store and injects them into the prompt. The LLM then answers grounded in those passages.

Key insight: The LLM's weights are never changed. RAG is purely an input-enrichment strategy.

Why not just put everything in the prompt?

You could paste an entire book into the context window. But:

RAG
  • Searches millions of documents in milliseconds
  • Costs almost nothing per query (only the relevant chunks use tokens)
  • Can be updated instantly — add or delete documents without retraining
Full-context stuffing
  • Limited by context window (e.g. 128K tokens ≈ one long book)
  • Expensive — you pay per token, every query
  • LLMs struggle to attend to info in the middle of very long contexts ("lost in the middle" effect)

The RAG Pipeline — Step by Step

Phase 1: Indexing (offline, done once)

1
Chunk the documents. Split each source (PDF, webpage, database row) into small, overlapping passages — typically 200–500 tokens each. Overlap ensures no information falls between the cracks.
2
Embed each chunk. Pass every chunk through an embedding model (e.g. OpenAI's text-embedding-3-small or open-source bge-base). This converts text into a dense vector — a list of, say, 768 floating-point numbers that encodes the meaning of the passage.
3
Store in a vector database. Save every (vector, chunk_text, metadata) triple in a specialised database optimised for nearest-neighbour search (e.g. FAISS, Pinecone, Weaviate, Chroma, Qdrant).

Phase 2: Retrieval + Generation (online, every query)

4
Embed the query. The user's question is passed through the same embedding model, producing a query vector in the same space.
5
Similarity search. The vector DB finds the K stored chunks whose vectors are closest to the query vector (usually via cosine similarity or dot product). This is blazingly fast — approximate nearest-neighbour algorithms like HNSW handle billions of vectors in <50ms.
6
Augment the prompt. The retrieved chunks are prepended to the user's question as context: 
System: Answer based ONLY on the following context.

Context:
[Chunk 1 text here]
[Chunk 2 text here]
[Chunk 3 text here]

User: What is the capital of Assyria?
7
Generate. The LLM reads the enriched prompt and produces a grounded answer. Because the relevant facts are right there in the context, hallucination is dramatically reduced.

How Data Is Stored: Vector Databases

A vector database is not a traditional SQL database. It doesn't search by keyword matching. Instead, it indexes high-dimensional vectors so that semantic similarity queries are fast.

What's actually stored?

Each entry (called a "point" or "document") consists of:

Field Example Purpose
id doc_0042 Unique identifier
vector [0.023, -0.841, 0.117, …] 768-dim embedding — the searchable "meaning"
text "The Transformer architecture was introduced in 2017…" Original chunk, returned to the LLM
metadata {source: "vaswani2017", page: 3} Filtering, citations

How is search so fast?

Brute-force comparison against millions of vectors would be too slow. Vector DBs use Approximate Nearest Neighbour (ANN) algorithms:

  • HNSW (Hierarchical Navigable Small World): builds a multi-layer graph where each node is a vector. Searching "hops" through the graph, narrowing in on the target region — like navigating a city by jumping between landmarks. Most popular in production.
  • IVF (Inverted File Index): clusters vectors into Voronoi cells at index time. At query time, only a few nearby cells are searched.
  • Product Quantization: compresses vectors into compact codes, trading a tiny bit of accuracy for massive memory savings.

Result: querying 1 billion vectors in ~50ms on a single machine.

How does this demo work?

Each "document" and your query are mapped to a tiny 3D vector using keyword heuristics (a stand-in for a real embedding model). The system computes cosine similarity between your query vector and every document vector, then ranks them. In production, the embedding model is a neural network (like BERT or a specialised retriever) and the vectors have hundreds of dimensions.

RAG vs. In-Context Learning

In-Context Learning (ICL) is when you put examples or instructions directly in the prompt and the LLM "learns" the task on the fly — no weight updates, just pattern-matching over the provided tokens.

RAG and ICL are complementary, not competing. In fact, RAG feeds into ICL: the retrieved documents become the in-context examples the model reasons over.

In-Context Learning
  • The model reads examples/instructions in the prompt
  • Limited to what fits in the context window
  • The human (or a program) decides what to include
  • Works for any task: translation, code, reasoning…
RAG
  • An automated retriever selects what to put in the prompt
  • Can search over millions of documents — far beyond any context window
  • Specialised for knowledge-grounded Q&A
  • The LLM still uses ICL to reason over the retrieved chunks
Why do they seem different?

ICL relies on whatever the user manually puts in the prompt. RAG automates the selection using vector search. The LLM itself doesn't know the difference — it just sees tokens in its context window either way. The magic of RAG is in the retrieval step before the LLM runs.

RAG and Long Context Windows

Modern LLMs have ever-growing context windows: 128K tokens (GPT-4), 200K (Claude), even 1M+ (Gemini). So why bother with RAG?

RAG can search far more than any context window

Approach Searchable data Cost per query
128K context window ~1 book $$$ (pay for all 128K tokens)
1M context window ~8 books $$$$ (massive token bill)
RAG (vector DB) Millions of documents $ (only retrieved chunks use tokens)

The "Lost in the Middle" Problem

Research shows that LLMs pay the most attention to the beginning and end of long contexts, often ignoring information buried in the middle. Even if a 200K-token window can hold your data, the model may fail to use it. RAG sidesteps this by only injecting the 3–5 most relevant chunks — keeping them front-and-center where attention is strongest.

In practice, RAG + long context work together: RAG retrieves the top 10–20 chunks, and the long context window gives the model room to reason over all of them at once.

Limitations & Failure Modes

  • Garbage in, garbage out. If the document store contains wrong information, RAG will confidently retrieve and present it.
  • Chunking matters. If a critical fact is split across two chunks and neither chunk is retrieved, the answer will be incomplete.
  • Embedding blind spots. Embedding models can misjudge similarity — a query about "Python" (the snake) might retrieve docs about "Python" (the language).
  • No reasoning over absence. RAG can't tell you "this information doesn't exist in the database." It will retrieve the closest thing, which may be irrelevant.
  • Latency. The retrieval step adds ~50–200ms per query. For real-time applications, this matters.

Summary

Question Answer
What is RAG? Fetch relevant docs → inject into prompt → LLM answers grounded in evidence.
How is data stored? As dense vectors in a vector DB (FAISS, Pinecone, etc.), indexed with ANN algorithms like HNSW.
How is data found? Cosine similarity between query embedding and stored embeddings. Top-K results are returned.
RAG vs. ICL? ICL = manual prompt engineering. RAG = automated retrieval that feeds ICL. They're complementary.
RAG vs. long context? RAG searches millions of docs cheaply. Long context holds ~1 book expensively. Best used together.
Are the LLM's weights changed? No. RAG only changes the input. The model itself is untouched.
