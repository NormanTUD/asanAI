The Core Question

Everyone agrees that modular programming is good. But how should you divide a system into modules?

Parnas observed that while programmers universally agreed on the benefits of modularity, almost nothing was said about the criteria for deciding where to draw the boundaries between modules. The conventional wisdom was simply: "follow the flowchart."

Expected Benefits of Modular Programming

👥

Managerial

Separate groups work on each module with little need for communication, shortening development time.

🔄

Flexibility

Drastic changes to one module should not require changes to others.

🧠

Comprehensibility

It should be possible to study the system one module at a time.

Two Decompositions Compared

❌ Conventional: Based on Processing Steps

Each module corresponds to a step in the flowchart. Modules share data structures and formats.

✓ Unconventional: Based on Information Hiding

Each module hides a design decision behind an abstract interface. Modules communicate through function calls, not shared data.

The Information Hiding Principle

The key criterion: Every module hides a design decision that is likely to change. The module's interface reveals as little as possible about its inner workings.

The fundamental insight: It is almost always incorrect to begin decomposition based on a flowchart. Instead, begin with a list of difficult design decisions or decisions likely to change. Each module is then designed to hide such a decision from the others.

What Should Be Hidden?

  • Data structures & access procedures — Internal linkings, storage formats, and modification procedures belong to a single module.
  • Calling sequences — The sequence of instructions to call a routine and the routine itself are part of the same module.
  • Control block formats — Formats used in queues and scheduling should be hidden within a control block module.
  • Character codes & orderings — Alphabetic orderings and encoding schemes should be encapsulated.
  • Processing sequences — The order in which items are processed should be hidden within a single module.

Hierarchical Structure

The information-hiding decomposition naturally produces a hierarchical structure (in the sense of Dijkstra's "THE" system). The "uses" relation forms a partial ordering.

Hover over modules to see their dependencies. Lower levels can be reused independently.

Two independent properties: Hierarchical structure and clean decomposition are both desirable but independent properties. You could have a hierarchy with bad decomposition (shared design decisions in interfaces), or good decomposition without a strict hierarchy.

Benefits of Hierarchy

🧱

Simplification

Higher levels are simplified because they use services of lower levels.

✂️

Pruning

Upper levels can be cut off, leaving a usable product (e.g., Line Storage can serve other applications).

🔁

Reuse

Lower modules can be the foundation for entirely different systems built on the same trunk.

Coupling Visualization

The two decompositions differ dramatically in inter-module coupling. Decomposition 1 has complex, format-dependent interfaces. Decomposition 2 has simple, abstract interfaces.

Key Takeaways

✗
DON'T: Decompose based on a flowchart (processing steps as modules).
✓
DO: Begin with a list of difficult design decisions or decisions likely to change. Each module hides one such decision.
🔄
REUSE: A good decomposition based on information hiding produces modules that carry over from one project to another — as demonstrated by Parnas's compiler/interpreter example where the same decomposition served both.

"Since, in most cases, design decisions transcend time of execution, modules will not correspond to steps in the processing."

— D.L. Parnas, 1972

Legacy & Modern Relevance

This 1972 paper laid the groundwork for nearly every modern software design principle:

📦

OOP Encapsulation

Classes hide implementation details behind public interfaces — directly from information hiding.

🏗️

Microservices

Each service hides a business capability behind an API — the same principle at system scale.

🔌

APIs & SDKs

Abstract interfaces that hide implementation — exactly what Parnas prescribed.

🧪

SOLID Principles

Single Responsibility, Open/Closed, Interface Segregation — all echo information hiding.
