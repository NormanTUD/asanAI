⏱️ Geschwindigkeit: Wo wird das Paket verarbeitet?

Je früher ein Paket abgefangen wird, desto weniger Overhead entsteht. BPF/XDP kann Pakete direkt am Netzwerktreiber verarbeiten – noch bevor der volle Netzwerkstack durchlaufen wird.

ufw (→ iptables → Netfilter) ~3.2M pps
iptables (→ Netfilter) ~3.5M pps
BPF/XDP (direkt am Treiber!) ~24M+ pps 🚀

⚖️ Detailvergleich

🟡

ufw

Uncomplicated Firewall
  • 📍 Läuft im Userspace
  • 🎯 Frontend für iptables
  • 🧠 Sehr einfach zu lernen
  • ⚡ Geschwindigkeit: Normal
  • 🔧 ufw allow 22/tcp
  • 📦 Nur Paketfilterung
  • 🐧 Ubuntu/Debian Standard
🟠

iptables

Netfilter Userspace Tool
  • 📍 Konfiguriert Kernel (Netfilter)
  • 🎯 Direkte Netfilter-Kontrolle
  • 🧠 Mittlere Lernkurve
  • ⚡ Geschwindigkeit: Gut
  • 🔧 Chains: INPUT, OUTPUT, FORWARD
  • 📦 Filter + NAT + Mangle
  • 🐧 Linux Standard seit ~2001
🔵

BPF / eBPF

Berkeley Packet Filter (extended)
  • 📍 Läuft IM Kernel direkt
  • 🎯 Programmierbare VM im Kernel
  • 🧠 Hohe Lernkurve (C/Rust)
  • ⚡ Geschwindigkeit: Extrem! 🚀
  • 🔧 XDP, tc, socket filter...
  • 📦 Filter + Tracing + Security + mehr
  • 🐧 Modern (Kernel 3.18+, voll ab 4.x)

🧪 BPF Playground

Hover über den Code um zu verstehen, was passiert

So sieht ein BPF-Programm aus (vereinfacht)

Dies ist ein XDP-Programm, das HTTP-Traffic durchlässt und alles andere droppt

// BPF-Programm: Einfacher Paketfilter
💡 BPF-Programme werden in C oder Rust geschrieben und dann in BPF-Bytecode kompiliert
#include <linux/bpf.h>
📚 BPF-Header aus dem Linux Kernel – definiert alle BPF-Strukturen und Helfer
#include <linux/if_ether.h>
🌐 Ethernet-Header-Definitionen – damit wir Pakete parsen können

SEC("xdp")
⚡ SEC("xdp") = Dieses Programm wird am XDP-Hook ausgeführt – das ist der FRÜHESTE Punkt, an dem ein Paket abgefangen werden kann, direkt am Netzwerktreiber!
int xdp_filter(struct xdp_md *ctx) {
🎯 Die Hauptfunktion! ctx enthält einen Pointer auf die rohen Paketdaten – kein Socket, kein Overhead
  void *data = (void *)(long)ctx->data;
📦 Direkter Zugriff auf die Paketdaten im Speicher – kein Kopieren, kein Buffering!
  void *data_end = (void *)(long)ctx->data_end;
📏 Ende der Paketdaten – BPF prüft IMMER Bounds, damit kein Buffer Overflow passiert (Verifier!)

  // Parse Ethernet + IP + TCP Header
🔍 Wir parsen die Header manuell – volle Kontrolle über jedes Byte!
  if (tcp->dest == htons(80))
🎯 Prüfe: Ist der Zielport 80 (HTTP)? htons() konvertiert die Byte-Reihenfolge
    return XDP_PASS; // ✅ Durchlassen!
✅ XDP_PASS = Paket weiterleiten an den normalen Netzwerkstack. Andere Optionen: XDP_DROP, XDP_TX (zurücksenden), XDP_REDIRECT

  return XDP_DROP; // 🚫 Alles andere droppen!
🚫 XDP_DROP = Paket sofort verwerfen, BEVOR es den Kernel-Netzwerkstack erreicht. Extrem effizient für DDoS-Schutz!
}
🏁 Das war's! Dieses Mini-Programm wird vom BPF-Verifier geprüft (keine Endlosschleifen, keine illegalen Speicherzugriffe) und dann JIT-kompiliert für native Geschwindigkeit.

🆚 Vergleich: Gleiche Aufgabe, drei Wege

Aufgabe: "Erlaube nur HTTP (Port 80), blockiere alles andere"

ufw

$ ufw default deny incoming
$ ufw allow 80/tcp
$ ufw enable
// 3 Befehle. Fertig. ✨

iptables

$ iptables -P INPUT DROP
$ iptables -A INPUT -p tcp \
  --dport 80 -j ACCEPT
$ iptables -A INPUT -m state \
  --state ESTABLISHED -j ACCEPT
// Mehr Kontrolle, mehr Tipparbeit

BPF/XDP (vereinfacht)

// ~30 Zeilen C-Code
// + Kompilieren mit clang
// + Laden mit bpftool/ip
$ ip link set dev eth0 \
  xdpgeneric obj filter.o
// Komplex, aber: 10x schneller! 🚀

🧠 Das Besondere an BPF: Nicht nur Firewall!

BPF ist eine universelle Programmierplattform im Kernel. Paketfilterung ist nur EIN Anwendungsfall:

🔥 XDP – Express Data Path

Pakete filtern/umleiten direkt am NIC-Treiber. DDoS-Schutz bei Cloudflare, Facebook.

🔍 Tracing & Profiling

Kernel-Funktionen beobachten ohne Neustart. Tools: bpftrace, perf.

🛡️ seccomp-BPF

Syscalls einschränken. Docker/Chrome nutzen das für Sandboxing!

🐳 Cilium (Kubernetes)

Ersetzt iptables komplett für Container-Networking. Schneller, skalierbarer.
